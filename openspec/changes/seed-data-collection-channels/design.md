## Context

「数据管理」模块已完成从「数据采集」+「事件查询」+「数据管理」三个独立页面的整合重构。核心数据模型调整为：
- `EventSource`（数据源，父容器）：定义分类、调度策略，每条代表一个数据领域（如 A股个股信息、国际新闻）。
- `DataChannel`（渠道，子端点）：绑定到某个 `data_source_id`，定义具体的采集方式（API/RSS/爬虫/AkShare）、端点地址、配置参数。

当前系统中种子数据文件 `seed_data_defaults.py` 已包含基础结构，但在实际启动后，前端页面可能出现数据展示异常、API 返回格式不匹配、调度器无法正确识别渠道等问题。本变更旨在通过完善种子数据、修复前后端数据链路，使数据采集功能达到「开箱即可验证」的状态。

## Goals / Non-Goals

**Goals:**
- 每个数据源下至少有一条真实可用的采集渠道种子数据。
- 数据源列表 API 正确返回所有内置数据源（含渠道嵌套展示）。
- 渠道管理 API 适配新的模型字段（`collection_method`, `endpoint`, `data_source_id`）。
- 前端页面能正确展示和操作数据源/渠道。
- 采集监控页面能展示基于渠道的聚合监控数据。
- 调度器能正确读取已启用渠道并执行采集任务。
- 所有相关测试通过。

**Non-Goals:**
- 不修改事件因子的核心计算逻辑。
- 不引入新的外部采集依赖（仅使用现有 AkShare / yfinance / RSS 等）。
- 不修改股票行情、K线等已有核心数据接口。

## Decisions

### 1. 种子数据覆盖范围
**决策**：为 6 大数据源类别各配置 1-2 条真实渠道：
- A股个股信息：东方财富新闻/公告、实时行情、财务数据
- 港股信息：港股新闻、港股行情
- 国际新闻：国际市场指数
- 财经资讯：新浪财经 RSS、华尔街见闻 API
- 板块数据：东方财富板块资金流（仅采集启用板块）
- 宏观数据：中国宏观经济指标

**理由**：覆盖用户最常用的数据类型，确保每个前端 Tab 都有可展示的数据。

### 2. 数据源列表返回逻辑
**决策**：`GET /api/event-sources` 返回所有 `is_builtin = 1` 的数据源，不再过滤。
**理由**：数据源是容器概念，所有内置数据源都应展示给用户，用户通过展开查看关联渠道并启用/停用。

### 3. 采集监控聚合维度
**决策**：监控列表以 `DataChannel` 维度聚合任务统计（而非 `EventSource`）。
**理由**：用户关心的是「某个渠道」的采集情况，渠道是实际执行采集的实体。

### 4. 测试策略
**决策**：先通过 `docker-compose up --build` 启动完整环境，然后逐一验证前端页面 → 后端接口 → 调度任务日志。
**理由**：种子数据的正确性需要在真实运行环境中验证，单元测试无法覆盖前端展示和定时调度。

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 种子数据中某些外部 API 在当前网络环境下不可用 | 渠道默认 `enabled=0`，用户可手动启用验证；采集失败只记录日志不影响系统启动 |
| 前端字段名与后端响应不匹配导致展示异常 | 逐一核对 `api.ts` 类型定义与实际 API 返回字段 |
| 调度器读取渠道时因模型变更导致异常 | 在 `run_news_collection_job` 中增加 `enabled` 检查并添加异常捕获 |
| 板块采集只采启用板块可能影响现有测试 | 更新测试用例以创建启用的板块记录 |

## Migration Plan

1. 更新 `seed_data_defaults.py` 确保种子数据完整且字段正确。
2. 修改 `src/api/events.py` 调整数据源列表返回逻辑。
3. 修改 `src/api/data_channels.py` 适配新模型字段。
4. 修改前端组件确保字段绑定正确。
5. 运行 `docker-compose up -d --build` 验证。
6. 修复验证过程中发现的问题。
7. 运行前后端测试套件，修复失败用例。

## Open Questions

- 采集监控的任务详情是否需要展示原始采集内容（如新闻全文）？当前设计在任务详情中展示关联事件列表。
