## Context

当前系统中「数据采集」、「事件查询」和「数据管理」作为三个独立页面存在，分别位于不同的导航入口。用户需要在多个页面之间跳转才能完成与数据相关的操作。同时，「数据源列表」页面混合展示了系统内置的基础数据源（如行情数据）和外部采集渠道（如东方财富新闻、公告），导致概念边界模糊。采集监控页面仅展示扁平列表，无法按采集源或渠道进行下钻查看。

## Goals / Non-Goals

**Goals:**
- 统一「数据管理」模块的入口和导航，将「数据采集」、「事件查询」和「数据管理」整合为同一模块下的子功能。
- 建立清晰的「数据源」与「渠道管理」概念边界：数据源为系统内置能力，渠道管理为外部采集入口。
- 渠道管理支持按「股票信息」和「国际事件」分类，支持激活/停用操作。
- 采集监控支持按时间、分类（自动/人工）、采集源、渠道等多维度筛选。
- 采集监控支持树形层级详情查看：采集源 → 渠道 → 具体采集数据。

**Non-Goals:**
- 不修改事件因子的核心计算逻辑（情感分析、去重、聚合等）。
- 不替换现有的 AkShare / yfinance 数据采集底层实现。
- 不引入新的外部数据源或采集渠道（仅对现有渠道进行重新分类和组织）。
- 不修改股票行情、K线、自选股等核心数据接口的行为。

## Decisions

### 1. 模块整合方式：Tab 切换而非页面合并
**决策**：将「数据采集」、「事件查询」和「数据管理」整合到「数据管理」大模块下，以 Tab 形式（如「数据源」、「渠道管理」、「事件查询」、「采集监控」）组织子功能，而非将三个页面的内容完全合并到一个页面。
**理由**：三个页面内容量较大，完全合并会导致页面过于复杂。Tab 切换可以保持各子功能的独立性，同时减少导航层级。
**替代方案**：侧边栏二级菜单 —— 考虑到 Ant Design 的导航习惯，Tab 在内容关联性强的场景下更直观。

### 2. 数据源与渠道的数据模型分离
**决策**：在数据库层面保留 `data_sources` 和 `channels`（或 `event_sources`）两个概念独立的表。`data_sources` 仅保留系统内置数据源（行情、K线等）；原有的外部采集类配置迁移到 `channels` 表，并新增 `category` 字段（`stock_info` / `global_event`）。
**理由**：清晰的概念边界便于后续扩展。若未来引入新的外部采集渠道，只需扩展渠道表，不影响内置数据源的定义。
**替代方案**：单表增加 `type` 字段区分 —— 会导致表职责不清，且两种类型的配置字段差异较大（渠道有 schedule、enabled 等，数据源没有）。

### 3. 渠道激活机制：软删除 + 状态字段
**决策**：渠道支持通过 `enabled` 布尔字段进行激活/停用，调度器在运行采集任务前检查该字段。不停用则跳过。
**理由**：简单直接，与现有 `event_sources.enabled` 字段兼容。无需物理删除即可控制采集行为。
**替代方案**：物理删除渠道记录 —— 会导致历史采集日志失去关联，不可接受。

### 4. 采集监控树形数据：后端聚合 + 前端 Tree 组件
**决策**：采集监控的树形详情（采集源 → 渠道 → 数据）由后端按层级聚合后返回嵌套 JSON 结构，前端使用 Ant Design 的 `Tree` 或嵌套 `Table` 组件展示。
**理由**：减少前端数据处理的复杂度，保证大数据量下的分页和筛选逻辑在后端完成。
**替代方案**：前端扁平数据自行组装树形结构 —— 在数据量大时性能差，且与后端分页冲突。

### 5. 采集监控筛选：后端 Query Param 组合筛选
**决策**：采集监控的筛选条件（时间范围、分类、采集源、渠道）通过 GET Query Parameters 传递，后端使用 SQLAlchemy 动态构建 WHERE 子句。
**理由**：筛选条件为简单的等值/范围查询，无需复杂的搜索 DSL。与现有 FastAPI 分页查询模式保持一致。

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 模块整合可能导致现有用户不习惯新的导航路径 | 在导航菜单保留短暂的「数据采集」别名重定向，或提供首次使用的引导提示；同步更新用户文档 |
| 数据源迁移至渠道管理时，历史配置数据需要迁移 | 编写 Alembic 数据迁移脚本，将现有 `event_sources` 中属于外部采集的记录归类到新的渠道分类下 |
| 树形监控数据量大时后端聚合性能下降 | 对采集监控接口增加分页和懒加载：先返回采集源列表，展开时再加载该源下的渠道和数据 |
| 渠道停用后，关联的定时调度任务可能报错 | 调度器在执行采集前统一检查 `enabled` 状态，停用的渠道直接跳过，不记录为失败 |

## Migration Plan

1. **数据库迁移**：创建 Alembic migration，为 `event_sources`（或新建 `channels`）表新增 `category` 字段，并迁移现有数据到正确分类。
2. **后端接口**：新增/修改渠道管理 CRUD 接口、采集监控筛选和树形详情接口；保持原有行情数据接口不变。
3. **前端页面**：
   - 新建「数据管理」主页面，包含 Tab 切换。
   - 重构「数据源列表」页面，移除外部采集类数据。
   - 重构「渠道管理」页面，增加分类和激活功能。
   - 重构「采集监控」页面，增加筛选和树形详情。
   - 调整 `Layout.tsx` 导航菜单。
4. **文档更新**：合并/重写 `client/public/docs/` 中「数据采集」、「事件查询」、「数据管理」相关 Markdown 文件。
5. **验证**：运行 E2E 测试覆盖数据管理模块的完整用户流程。

## Open Questions

- 是否需要在渠道管理中支持自定义新增渠道（用户自行配置新的采集源），还是仅对现有内置渠道进行分类管理？
- 「数据管理」Tab 的默认展示页面是「数据源」还是「采集监控」？
- 树形监控详情中「具体采集数据」的粒度是什么（单条事件/新闻/公告，还是按批次聚合）？
