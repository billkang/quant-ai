## Context

当前 `Layout.tsx` 采用经典的「侧边栏 + 主内容区」两栏布局。侧边栏底部（bottom area）同时放置了：
1. 告警入口（铃铛图标 + Badge）
2. 主题切换器（Dropdown）
3. 用户信息（用户名 + 登出 Dropdown）

这三个元素挤在 220px 宽的侧边栏底部，随着未来功能增加，空间愈发局促。同时，主内容区顶部没有任何全局操作区，形成了「所有控制都在左侧，右侧大片空白」的不平衡视觉。

## Goals / Non-Goals

**Goals:**
- 在主内容区顶部新增一条固定 header bar（高度约 56-64px）
- 将告警、主题切换、用户信息三个控件从侧边栏底部迁移至 header 右侧
- 保持侧边栏纯粹作为导航用途，底部区域清空或仅保留版权信息
- 各控件在 header 中的排列顺序：告警 → 主题切换 → 用户信息
- 响应式：小屏幕下 header 控件保持可见，必要时可收缩

**Non-Goals:**
- 不修改任何后端 API 或数据模型
- 不修改三个控件自身的内部逻辑（仅移动位置）
- 不引入新的全局控件（搜索栏、面包屑等本次不涉及）
- 不修改侧边栏导航项的内容或顺序

## Decisions

### 1. Header 实现方式：Layout 内部新增 div，不拆分为独立组件
- **决策**：在 `Layout.tsx` 内部直接新增一个 `<header>` 元素，不拆分为 `Header.tsx` 独立组件。
- **理由**：header 逻辑简单（仅展示三个现有控件），独立组件会增加文件数量和维护成本。如果未来 header 复杂度显著增加（如加入全局搜索、面包屑、多 tab），再拆分不迟。
- **替代方案**：创建 `Header.tsx`。拒绝原因：过度拆分，当前三个控件都是已有实现，仅需改变渲染位置。

### 2. Header 固定定位：sticky 而非 fixed
- **决策**：header 使用 `position: sticky`（或直接在 flex 布局中作为普通块级元素），不占用 `fixed` 层级。
- **理由**：当前 Layout 的主内容区已经是 `marginLeft: 220` 的普通文档流，header 作为 `<main>` 的第一个子元素即可自然置顶，无需 fixed。滚动时 header 随内容一起滚动（不悬浮），减少 z-index 冲突。
- **替代方案**：fixed header + main padding-top。拒绝原因：增加复杂度，且当前设计不需要悬浮 header。

### 3. 侧边栏底部清空后的处理：移除整个 bottom div 的复杂布局
- **决策**：侧边栏底部区域彻底清空，移除现有的 flex bottom div（包含告警、主题、用户）。
- **理由**：清空后侧边栏底部更简洁，导航视觉更聚焦。如果未来需要版权信息，可在 sidebar 最底部添加极简的文本。
- **替代方案**：保留底部区域，仅移除三个控件。拒绝原因：留下空白区域无意义，不如彻底简化。

### 4. Header 视觉风格：与 sidebar 同色系的 surface 背景
- **决策**：header 背景色使用 `var(--bg-surface)`，底部加 1px `var(--border)` 分隔线，与侧边栏形成统一视觉语言。
- **理由**：保持现有暗色主题的一致性，header 作为内容区的「顶栏」而不是独立悬浮元素。

## Risks / Trade-offs

- **[Risk]** 移动后用户可能需要短暂适应新位置
  → **Mitigation**：三个控件的图标和交互完全一致，用户学习成本极低。
- **[Risk]** 移动端（小屏幕）下 header 右侧控件可能拥挤
  → **Mitigation**：当前项目主要面向桌面端量化分析，移动端不是重点。如后续需要，可将三个控件收入一个「更多」Dropdown。
- **[Risk]** 测试中的 DOM 选择器失效
  → **Mitigation**：修改测试中的查询方式，从「侧边栏底部查找」改为「header 区域查找」。

## Migration Plan

无需迁移。纯前端 UI 调整，不涉及数据或 API。部署后用户刷新页面即可看到新布局。
